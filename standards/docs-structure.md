# 작업문서 표준 구조

모든 프로젝트가 같은 자리에 같은 문서를 둔다. 에이전트는 어떤 프로젝트를 열어도 이 구조를 기대하고 읽는다.

```
<프로젝트>\
├── CLAUDE.md          ← 에이전트 지침, 1페이지 (CLAUDE-template.md 참조)
├── README.md          ← 공개용: 문제 → 해결 → 스택 → 실행법 → 스크린샷
└── docs\
    ├── PLAN.md        ← 계획·결정 로그
    ├── DEVLOG.md      ← 작업일지 (최신이 위)
    ├── EXPERIENCE.md  ← 마무리 시 경험블록 공개용 축약본
    └── adr\           ← 설계 결정 기록 (arch-budget 스킬이 생성)
```

## 각 문서의 역할과 포맷

### PLAN.md — 무엇을 왜 하기로 했는가
- 상단: 목표 한 줄 + 현재 단계.
- 본문: 결정 로그 (최신이 위). 항목마다 **결정 / 근거 / 기각한 대안 1줄**.
- 아키텍처 결정은 여기 말고 `docs/adr/`에. PLAN.md에는 링크만.
- 기획 전략·컨셉 상세도 여기에 — CLAUDE.md에 넣지 않는다 (CLAUDE.md는 1페이지).

### DEVLOG.md — 무슨 일이 있었는가 (최신이 위)
```markdown
## YYYY-MM-DD
- 한 일:
- 막힌 것 → 해결: (증상과 원인을 구체적으로 — 나중에 경험블록 재료가 된다)
- 다음:
```
- 진행상황·자잘한 결정은 메모리가 아니라 여기에 적는다 (메모리 규칙 참조).

### EXPERIENCE.md — 이 프로젝트에서 무엇을 얻었는가
- **원본은 커리어 허브 `_career\experience-blocks\<프로젝트>.md`** (experience-block 스킬로 작성, 면접용 STAR 포함).
- 프로젝트의 EXPERIENCE.md에는 공개 가능한 축약본만: 개요 / 문제 / 해결 / 성과. 같은 내용을 두 곳에 전부 복제하지 않는다.

## 규칙

- 문서가 없는 상태로 코드부터 쌓지 않는다. 최소한 CLAUDE.md(상태·목적 분류 포함)는 프로젝트 시작 시점에 만든다.
- 루트에 md 파일을 늘리지 않는다 — 문서는 docs\ 아래로. 예외는 CLAUDE.md, README.md, LICENSE.
- 프로젝트 상태가 바뀌면(활성→완료 등) CLAUDE.md의 상태 줄부터 갱신한다.
