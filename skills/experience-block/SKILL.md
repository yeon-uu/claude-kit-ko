# experience-block — 경험블록 기록

> 출처 합성: Julia Evans "Brag Documents"(기록 습관·정확성 원칙) + Paramchoudhary/ResumeSkills(X-Y-Z 공식, 6대 지표, 발굴 질문) + ARPeeketi/claude-resume-kit(출처 플래그).
> 포맷이 고정이라 어떤 모델이든 작성 가능. 숫자 발굴 인터뷰는 대화가 필요하므로 자동 생성하지 말 것.

## 원칙

- **실제보다 좋게 말하지 말고, 정확히 그만큼 좋게** (jvns). 과장 동사("혁신했다", "주도했다" 남발) 금지.
- 모든 수치에 출처 플래그: `[측정]`(로그·계측값) / `[추정]`(근거 있는 계산) / `[내부]`(팀 자체 평가). 플래그 못 붙이는 숫자는 쓰지 않는다.
- 프로젝트가 끝난 뒤 몰아 쓰지 말고 마일스톤마다 갱신한다. 기억은 2주를 못 간다.

## 파일 구조

```
_career\experience-blocks\<프로젝트명>.md   ← 프로젝트당 1파일, 블록 여러 개 가능
```

## 블록 포맷 (고정)

```markdown
## <블록 제목 — 한 줄 성과 요약>
- 기간: YYYY-MM ~ YYYY-MM / 역할: <본인 역할, 팀이면 N명 중 담당>
- 키워드: <JD에 나오는 표현 그대로 태그 — 예: FastAPI, 비동기, Redis 캐싱, 시큐어코딩>

### 문제 (Challenge)
<누가 뭘 못 해서 곤란했는가, 제약 조건 포함 2~4문장>

### 해결 (Action)
<내가 한 일. 설계 결정과 기각한 대안 1개 포함>

### 성과 (Result)
<X-Y-Z: [Z]를 해서 [Y]로 측정되는 [X] 달성. 수치에 플래그>

### 배운 것
<한 줄. 다음에 다르게 할 것>

### 증거
<커밋/PR/배포 URL/수상 공문/발표 자료 — 면접에서 보여줄 수 있는 것>

### 면접용 STAR 풀버전
<위 내용을 말로 풀어 60~90초 분량. 이력서 한 줄 버전도 마지막에>
```

## 레포 스캔 모드 (레포 URL·로컬 경로가 입력일 때)

1. **수집** — 로컬 경로면 직접 읽고, URL이면 `gh repo view <url>` + `gh api`(README) 또는 얕은 clone. 확인할 것: README·docs/(문제 정의·아키텍처), `git log --oneline`(작업 흐름·기간), `gh pr list --state merged`(협업 흔적), 언어·스택 구성.
2. **초안 생성** — 위 재료로 블록의 키워드/기간/문제(Challenge)/해결(Action)까지 채운다. 커밋 이력에서 "막혔다가 고친 흔적"(fix 연속, 되돌림)을 찾으면 트러블슈팅 후보로 표시.
3. **인터뷰 (생략 금지)** — 성과 수치와 설계 이유는 레포에 없다. **자동으로 지어내지 말고** 6대 지표 질문 3~5개를 던져 받아 적는다. 답을 못 받은 칸은 `[미정]`으로 남기고 저장한다 — 빈칸이 날조보다 낫다.
4. **저장** — 블록 파일 + INDEX.md 한 줄 갱신. 초안 상태면 제목에 `(초안)` 표시.

## 면접 대비 모드 ("면접 준비해줘" 요청 시)

1. `INDEX.md`에서 대상 직무·회사와 관련된 블록 선별 (특정 회사면 `_career\jd-analysis\`의 해당 리포트 우선 참조).
2. 블록의 STAR 풀버전을 기반으로: 예상 질문(기술 깊이 + 꼬리질문 2단계까지) / 답변 포인트(수치·증거 링크 포함) / 약점 질문("왜 X 안 썼어요?" 류 — 기각한 대안이 답의 재료)을 정리.
3. 블록에 없는 걸 지어내지 않는다 — 준비 안 된 영역은 "여기는 갭"이라고 명시 (gap-plan 참조).

## 숫자 발굴 인터뷰 (성과가 "없다"고 할 때)

6대 지표로 캐묻는다 — 돈 / 시간 / 퍼센트 / 규모 / 품질 / 빈도:
- "몇 명이 썼나? 요청이 하루 몇 건?" (규모)
- "그 작업 전엔 얼마나 걸렸고 후엔?" (시간)
- "에러율·실패 건수가 어떻게 변했나?" (품질)
- "심사·제출까지 며칠 만에?" (빈도·속도)

리팩터링처럼 애매한 작업은 3단계로: 목표 → 구체 행동 → 관찰된 효과(간접 효과 포함).

## 학생·신입 특화 규칙

- CTF·대회 문제 풀이, 동아리 세미나 발표, 오픈소스 이슈 리포트도 블록 대상이다 — "업무"만 경험이 아니다.
- 수치가 빈약하면 규모·제약을 수치로 쓴다: "2GB 메모리 t3.small에서", "3인 팀 12일 안에".
- 블록 작성 후 `_career\experience-blocks\INDEX.md`에 한 줄 추가 (제목 + 키워드) — jd-fit 스킬이 이 인덱스를 검색한다.

## 글쓰기
