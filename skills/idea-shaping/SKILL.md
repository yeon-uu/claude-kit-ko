# idea-shaping — 아이디어 구상·검증

> 출처 합성: obra/superpowers(brainstorming)의 질문 규율·하드게이트 + anthropics/knowledge-work-plugins(product-brainstorming)의 Diverge→Provoke→Converge + bwerneckm/startup-skills의 점수제 판정. 공모전 모듈은 자체 설계.
> 절차와 점수표가 고정이라 어떤 모델이든 따를 수 있다. Provoke(반론 생성)의 질은 모델 역량에 비례 — 최종 판정은 Opus 이상 권장.

## 원칙

- 질문은 **한 번에 하나**, 가능하면 객관식.
- 사용자가 아이디어를 확정("이걸로 간다")하기 전까지 기획서·코드·파일 생성 금지 (하드게이트).
- 과잉 긍정 금지. 이 스킬의 목적은 응원이 아니라 **죽일 아이디어를 빨리 죽이는 것**이다.
- 첫 아이디어에 앵커링하지 말 것 — 발상 단계에서는 범위·방식·타이밍을 달리한 대안 3개 이상을 만든 후 좁힌다.

## 절차

### 0. 분기 — 첫 질문은 항상 목적 분류

"이 아이디어는 ① 공모전·논문·포폴용 ② 상업 출시용 ③ 연습·학습용 중 무엇인가?"
분기에 따라 아래 다른 트랙을 탄다. 둘 다라면 "1차 목적"을 고르게 한다.

### A. 공모전 트랙

1. **심사기준 확보** — 공고문의 심사 배점표를 요청. 없으면 웹에서 찾고, 그래도 없으면 "배점표 없이는 역산 불가"를 명시하고 일반 기준(적합성·창의성·완성도)으로 가정임을 표기.
2. **배점 역산** — 배점표의 각 항목 → 아이디어의 어떤 요소가 그 점수를 따는지 매핑 표 작성. 배점 높은 항목에 대응 요소가 없으면 그 자리에서 지적.
3. **역대 수상작 조사** — 같은 공모전(없으면 유사 공모전) 수상작 3~5개를 웹 검색으로 확인하고 출처 링크를 남긴다.
4. **차별화 축 명시** — "역대 수상작은 전부 X인데 이건 Y" 형태의 방어 논리를 한 문장으로 만들 수 있어야 통과. 못 만들면 레드오션 판정.
5. **실격 조건 체크** — 공고의 필수 요건·금지 사항 목록화 (데이터 필수 사용, 재사용 금지 등).

### B. 상업 트랙

1. **사전 필터 (25점, 18점 미만이면 검증 착수 보류)** — 각 1~5점: 문제 빈도 / 문제 강도 / 지불 의사 / 창업자(본인) 적합성 / 타이밍. 점수 근거를 한 줄씩 명시. 추정이면 "추정"이라고 쓴다.
2. **가장 위험한 가정(RAT)** — 의존 가정을 나열하고 중요도×확신도 2×2로 배치. "틀리면 죽는데 확신 없는" 가정 하나를 지목.
3. **싼 테스트 제안** — 그 가정을 일주일·0원 근처에서 검증할 방법 (랜딩페이지, fake door, 컨시어지, 수동 운영).
4. **레드오션 체크** — 동일 기능 기존 서비스를 검색해 링크로 제시. 있으면 "그럼에도 쓸 이유"를 답하게 한다.

### C. 공통 마무리

1. **Provoke** — 확정 직전에 반드시 묻는다: "이 아이디어에 대한 가장 강한 반론은 무엇인가?" 스스로 반론 2개를 만들어 제시하고 사용자의 응답을 듣는다.
2. **판정** — GO(그대로 진행) / PIVOT(각도 수정, 수정안 1~2개 제시) / KILL(근거와 함께). 결론 → 근거 → 트레이드오프 순.
3. **기록** — 사용자 승인 후 프로젝트 `docs/PLAN.md`에 결정·근거·기각 대안을 기록. 프로젝트 폴더가 없으면 기록 위치를 물어본다.

## 안티패턴

- 배점표·수상작 확인 없이 "좋은 아이디어네요"로 진행.
- 프레임워크 덤핑 (한 세션에 SCAMPER+JTBD+린캔버스 전부 꺼내기) — 필요한 것 하나만.
- 근거 없는 시장 수치 인용 — 수치는 출처 링크가 있거나 "추정" 표기가 있어야 한다.
