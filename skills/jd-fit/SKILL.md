# jd-fit — JD 적합성·갭 계획

> 출처 합성: Paramchoudhary/ResumeSkills(요구사항 3분류·가중 점수·갭 3등급) + santifer/career-ops(요구사항↔증거 매핑 표, 파일 트래커) + varunr89/resume-tailoring-skill(발굴 인터뷰, 사실 보존 원칙).
> 추출·점수는 규칙 기반이라 어떤 모델이든 가능. 갭 채움 프로젝트 제안은 Opus 이상 권장.

## 원칙

- **사실 보존**: 없는 경험을 만들어내지 않는다. 갭은 숨기지 말고 명시한 뒤 채울 계획을 만든다.
- 웹 검색은 5회 이내로 제한 (공고 유효성·회사 정보 확인용).

## 절차

### 0. 풀코스 모드 ("풀코스로 돌려줘", "지원 준비 전부 해줘", "쭉 뽑아줘")

공고 하나를 입력받아 **아래를 중단 없이 순서대로 전부 실행**하고 통합 리포트 1개로 뽑는다:

1. 적합성 분석 (§1~5) → 점수·강점/약점·갭 계획.
2. 점수 **50% 이상**이면 지원서 초안 (§5.7 지원동기 + 이력서 재배열)까지 진행. **50% 미만**이면 여기서 멈추고 갭 계획만 보고 (미달인데 지원서를 쓰는 건 시간 낭비 — 이유와 함께 판정).
3. interview-prep 스킬 절차로 예상질문 3종(일반·팩트형·경험) 생성.
4. **통합 리포트** 저장: `jd-analysis\YYYY-MM-DD-<회사>-풀코스.md` — 구성 고정:
   ① 결론 한 줄 (지원 권장 여부 + 점수) ② 강점 Top3 / 약점 Top3 ③ 요구사항↔증거 매핑 표 ④ 지원동기 초안 ⑤ 이력서 한 줄 재배열 ⑥ 예상질문 3종 ⑦ **남은 액션 체크리스트** (갭 채움 · 서류 마감일 · 팩트형 답 검증 · 30초 리허설)
5. `applications.md`에 한 줄 기록.

사용자 확인은 완료 후 한 번만 받는다 (톤 수정·팩트형 답 검증은 리포트 보고 나서). 소요는 웹 서치 포함 수 분 단위임을 시작할 때 고지.

### 1. 공고 유효성
마감일 지났는지, 상시채용인지 먼저 확인. 지났으면 분석 가치(유사 공고 대비용)를 물은 뒤 진행.

### 2. 요구사항 추출 — 언어 마커 기반 3분류
- **Required**: "자격요건", "필수", "~경험이 있으신 분" 아래 항목
- **Preferred**: "우대사항", "~면 더 좋아요" 아래 항목
- **도메인·소프트**: 산업 지식(금융 규제, 게임 등), 협업 표현
각 항목을 원문 표현 그대로 표에 옮긴다 (키워드 미러링을 위해 의역 금지).

### 3. 증거 매핑
`_career\experience-blocks\INDEX.md`와 각 블록을 읽고 요구사항별로 매핑:

| 요구사항 (원문) | 구분 | 보유 | 증거 (블록·링크) | 확신도 |
|---|---|---|---|---|
| | Required/Preferred | ●보유 / ◐부분 / ○없음 | | 높음/중간/낮음 |

- 블록에 없는데 있을 법한 경험은 **발굴 인터뷰**로 캐묻는다 ("동아리에서 비슷한 거 안 했나? 대회에서?"). 나오면 experience-block 스킬로 블록부터 갱신.

### 4. 적합성 점수
- Required 충족률 × 0.7 + Preferred 충족률 × 0.3 (◐는 0.5로 계산).
- 해석 밴드: **75% 이상** 지원 적극 권장 / **50~74%** 스트레치 — 지원하되 갭 보완 서술 필요 / **50% 미만** 현시점 미달 — 갭 계획 우선.
- 판정은 결론 → 근거 → 트레이드오프 순으로. 좋게 포장하지 않는다.

### 5. 갭 3등급 + 채움 계획
| 등급 | 의미 | 처리 |
|---|---|---|
| Critical | Required인데 ○없음 | 채우기 전 지원 비추천. 채움 수단 필수 제시 |
| Major | Required인데 ◐부분 | 자소서·면접에서 보완 서술 + 채움 수단 |
| Minor | Preferred ○/◐ | 인접 경험으로 커버 가능 표기 |

채움 수단은 구체적으로 — "Redis 경험 없음 → 기존 포트폴리오 프로젝트에 캐시 레이어를 얹는 1주 미니 프로젝트", "자격증 → 다음 회차 접수일 명시". 기간과 완료 판정 기준을 붙인다.

### 5.5 직무 일반 모드 (특정 공고 없이 직무명만 주어질 때)

1. 해당 직무의 **최근 공고 5~10개**를 웹 서치로 수집 (대기업·중견·스타트업 섞어서 — 한쪽만 보면 왜곡). 출처 링크 전부 기록.
2. Required 항목을 공고 횟수로 집계한 빈도표 작성: | 요구사항 | 등장 횟수/N | — 이것이 "그 직무가 통상 요구하는 것".
3. 빈도 상위 항목을 내 경험블록과 대조 → 통상 강점(●) / 통상 부족(○) 판정. 이후 갭 3등급·채움 계획은 동일 절차.
4. 리포트는 `jd-analysis\YYYY-MM-DD-<직무명>-일반.md`로 저장. 공고 수가 5개 미만이면 표본 부족을 명시.

**입력에 깃허브 프로필·레포 링크가 있으면**: 경험블록에 없는 증거 후보(핀 레포, 기여 이력)를 스캔하되, 새 증거를 찾으면 매핑에 바로 쓰지 말고 experience-block 스킬로 블록부터 갱신 — 블록이 증거의 원본이다.

### 5.7 지원서 초안 ("지원동기 써줘", "이력서 맞춰줘" 요청 시)

- **지원동기**: 기업 리서치(사업·기술 스택·최근 행보 — 출처 링크 필수) + 직무 연결(내 경험블록의 구체 스토리 1개를 요구사항에 연결) 2단 구조. 일반론("귀사의 발전에 기여") 금지.
- **이력서 맞춤**: 관련 블록들의 "이력서 한 줄"을 공고 키워드 등장 순서로 재배열하고, 공고 원문 표현으로 키워드 미러링.
- **사실 보존**: 블록에 없는 경험을 만들어 넣지 않는다. 문장은 write-human 스킬 기준.

### 6. 저장
- 리포트: `_career\jd-analysis\YYYY-MM-DD-<회사>-<직무>.md`
- `_career\gap-plan.md` 갱신: 갭별 채움 수단을 누적하되, 여러 JD에서 반복 등장하는 갭을 상단으로 (반복 등장 = 시장이 요구하는 것).
- `_career\applications.md`에 한 줄 트래킹 (날짜/회사/직무/점수/판정).
