# arch-budget — 자원 제약 아키텍처 설계

> 출처: obra/superpowers(brainstorming)의 설계 게이트 + anthropics/knowledge-work-plugins(architecture)의 제약 선언·평가 매트릭스 + affaan-m/ECC의 ADR 포맷을 소규모·단기 프로젝트용으로 합성.
> 절차·표 형식이 고정이라 어떤 모델(Sonnet 포함)이든 그대로 따르면 된다. 단, 대안 발상 자체는 판단이 필요하므로 중요한 설계는 Opus 이상 권장.

## 왜 이 절차인가

설계 없이 구현부터 시작하면 파일 간 불일치와 과잉 생성이 발생한다. 반대로 범용 베스트프랙티스를 그대로 쓰면 1인·단기·저사양 프로젝트에 과설계가 된다. 그래서 **제약을 먼저 숫자로 박고, 그 제약 안에서만 비교**한다.

## 절차 (순서 고정)

### 1. 제약 선언 — 설계 논의 전에 반드시 확정

사용자에게 없는 값만 물어라 (한 번에 하나, 객관식 우선):

| 항목 | 예 | 미확정 시 기본값 |
|---|---|---|
| 서버 자원 | EC2 티어 또는 "로컬만" | t3.small |
| 팀 | 인원 × 실투입 기간 | 1인 |
| 기한 | 마감일 | 물어볼 것 |
| 목적 분류 | 공모전·포폴용 / 상업용 / 연습용 | 물어볼 것 |
| 성능 목표 | 동시 사용자, 응답시간 | 공모전=시연 안 죽는 수준 |

EC2 티어 → 메모리 예산 환산 (이 표 기준으로 계산):

| 티어 | vCPU / RAM | 실제 앱 가용 메모리 (OS·Docker 제외) |
|---|---|---|
| t3.micro | 2 / 1GB | ~500MB — DB 컨테이너 동거 불가 |
| t3.small | 2 / 2GB | ~1.2GB — 앱+PG 동거 가능, Redis 추가는 계산 후 |
| t3.medium | 2 / 4GB | ~3GB |

### 2. 대안은 반드시 2~3개, 이름 붙여 비교

"방법이 하나뿐"이라는 결론 금지. 각 대안에 이름을 붙이고(예: "A. PG 단독", "B. PG+Redis") 아래 표로 비교한다:

| 대안 | 메모리 예산 내? | 기한 내 구현 가능? | 운영 부담 | 목적 적합성 | 판정 |
|---|---|---|---|---|---|

- **추천안을 표 아래 첫 문장에 명시**하고 근거 → 트레이드오프 순으로 서술.
- 공모전용이면 "심사 시연에서 죽지 않는가"가 확장성보다 우선. 상업용이면 운영·과금·백업이 평가축에 추가된다.

### 3. YAGNI 목록 — 안 만드는 것을 명시

설계안에 "이 제약에서 만들지 않는 것" 섹션을 반드시 포함한다 (예: 관리자 페이지, 통계 API, 멀티리전, MSA 분리). 나중에 필요해지는 조건도 한 줄씩 적는다 ("동시 사용자 100 넘으면 Redis 도입").

### 4. 승인 후 ADR 기록

사용자가 설계를 승인하면 `docs/adr/NNNN-<제목>.md`로 기록 (승인 전 파일 생성 금지):

```markdown
# ADR-NNNN: <결정 한 줄>
날짜: YYYY-MM-DD / 상태: accepted
## Context — 제약 포함 2~5문장 (티어·팀·기한 숫자 명시)
## Decision — 1~3문장
## 기각 대안 — 각각 이름 + 기각 사유 1줄
## Consequences — 긍정 / 부정 / 리스크
```

- `docs/adr/`에 인덱스가 없으면 README.md 표를 함께 생성.
- 사소한 결정(변수명, 라이브러리 마이너 버전)은 ADR 금지 — 2분 안에 읽을 가치가 있는 결정만.
- 기존 결정을 뒤집으면 새 ADR을 쓰고 옛 것을 `superseded by ADR-NNNN`으로 표기.

## 안티패턴

- "간단하니까 설계 생략" — 아무리 작아도 1·2번은 거친다 (표가 3줄이어도 된다).
- 제약 숫자 없이 "확장성 좋은 구조" 같은 형용사 비교.
- 승인 전에 코드/파일 생성 시작.
